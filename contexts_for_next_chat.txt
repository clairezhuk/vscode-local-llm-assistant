Ось структуроване резюме твого проєкту. Це чудова шпаргалка для розуміння поточної архітектури та планування наступних кроків.

### 🎯 Суть проєкту
Локальний AI-асистент програміста (аналог GitHub Copilot) у вигляді розширення для VS Code. 
* **Ресурси:** Оптимізовано під слабке залізо (2GB VRAM, 8GB RAM). Використовує квантовані моделі (наприклад, Qwen 2.5 Coder 1.5B) через `llama.cpp`.
* **Ключові фічі:** Чат із підтримкою Markdown, розуміння контексту з прикріплених файлів, автодоповнення коду (Ghost text) та здатність виконувати безпечні команди в інтегрованому терміналі.

---

### 📂 Файлова структура та відповідальність класів

**Фронтенд (VS Code Extension - TypeScript):**
* `chat_view.ts` (`ChatViewProvider`): Відповідає за інтерфейс чату (Webview). Рендерить Markdown через `marked.js`, обробляє введення тексту та прикріплення файлів.
* `ghost_text.ts` (`GhostTextProvider`): Реалізує автодоповнення. Передає на бекенд лише останні 15 рядків перед курсором (Sliding Window), щоб не перевантажувати контекст і забезпечити швидку відповідь.
* `client.ts`: API-клієнт для зв'язку з локальним бекендом.

**Бекенд (FastAPI - Python):**
* `llm.py` (`LLMEngine`): Обгортка навколо `llama_cpp`. Відповідає за безпосередню генерацію тексту та підрахунок токенів.
* `orchestrator.py` (`Orchestrator`): "Мозок" системи. Керує циклом мислення: переклад, маршрутизація, генерація, перевірка (Self-Correction) та виклик інструментів.
* `file_processor.py` (`FileProcessor`): Оптимізує контекст. Замість читання всього файлу, робить AST-парсинг для Python (витягує сигнатури), Regex-пошук для JS/C++ або коротку сумаризацію для тексту/JSON.
* `manager.py` (`ContextManager`): Зберігає історію чату та форматує системні промпти.
* `tools.py` (`ToolExecutor`): Виконує консольні команди.
* `routes.py`: FastAPI ендпоінти (`/chat` та `/completion`).

### ⚙️ Поточний стан: Логіка обробки промптів (Agentic Flow)

Наразі реалізовано багатокроковий конвеєр:

1.  **Препроцесинг (Транслятор):** Легкий запит до LLM, який виправляє одруківки та перекладає промпт англійською, жорстко забороняючи генерувати відповіді.
2.  **Маршрутизація (Intent Classifier):** Визначає тип задачі:
    * `1`: Теорія/Питання (віддає звичайний текст).
    * `2`: Написання коду (вимагає ізольованого Markdown-блоку та обов'язкового `return`).
    * `3`: Термінал (вимагає чистої команди). + *Запобіжник: перевіряє, чи починається команда з безпечного префікса (pip, npm, git тощо).*
3.  **Генерація та Рецензування (Self-Correction):** * Для Intent 2: Після генерації код проходить `_static_code_check` (через модуль `ast`). 
    * Перевіряється відсутність `SyntaxError`, наявність викликаних, але не визначених/імпортованих функцій (`NameError`), та наявність `return`.
    * Якщо є помилка, бекенд автоматично запускає цикл виправлення (до 5 спроб), надсилаючи помилку назад моделі.
4.  **Виконання (Для Intent 3):** Команда виконується в системі, і результат повертається у чат.

### 🧪 Система тестування (Benchmarks)

Автоматизована пісочниця для перевірки стабільності архітектури.

* **Структура (`tests/suites/*.json`):** Тести поділені на тематичні блоки (базові алгоритми, CLI, складні задачі). Кожен тест містить: `id`, `query`, `expected_intent`, `must_contain` / `must_not_contain` (ключові слова), та `execution` (assert-перевірки або bash-скрипти для верифікації).
* **Раннер (`tests/run_benchmark.py`):** * Створює ізольовані тимчасові папки (`tempfile`) для кожного тесту.
    * Зшиває згенерований LLM код із `assert`-перевірками та запускає його.
    * Оцінює метрики: `intent_ok` (чи вгадав маршрутизатор), `format_ok` (перевірка ключових слів) та `exec_ok` (чи пройшов код компіляцію і тести логіки).
    * Формує CSV-звіт і логує помилки (Bad intent, Bad format, Execution fail) у папку `results/logs/`.
    * Підтримує запуск конкретних сюїт і лімітування кількості тестів (`-n`).

### 🚀 Що далі покращувати (Roadmap)

Аналіз логів тестування показав, що 1.5B модель чудово знає синтаксис, але має "короткий горизонт логіки" (помиляється в задачах >2 кроків) та обмежений контекст. Наступні кроки:

1.  **Task Planner (Планувальник підзадач):** Для складних задач (Intent 2) додати проміжний крок. Спочатку модель генерує JSON-план кроків, а потім пише код для кожного кроку окремо. Це вирішить проблему алгоритмічної "каші" (як у тестах `find_unique` чи `reverse_vowels`).
2.  **Runtime Self-Correction (Динамічне налагодження):** Зараз цикл виправлення ловить лише синтаксис та відсутні імпорти. Треба навчити бекенд запускати згенерований код (в ізольованому середовищі) з тестовими даними користувача, щоб ловити `AssertionError` та `TypeError` *до* виводу результату в чат.
3.  **Пошук за RAG / TF-IDF:** Для великих проєктів AST-парсингу буде недостатньо. Потрібно розбити файли на чанки і реалізувати легкий локальний пошуковик, який підтягуватиме у промпт лише 2-3 релевантні функції замість усього дерева.
4.  **Покращення UI для контексту:** Додати можливість юзеру бачити, *які саме* шматки файлу були відправлені моделі, щоб краще розуміти, чому вона прийняла те чи інше рішення.
