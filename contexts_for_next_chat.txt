Project Essence

A local AI programming assistant built as a VS Code
extension.

  - Resources: Optimized for entry-level hardware (2GB VRAM, 8GB RAM). Utilizes
    quantized Qwen 2.5 Coder 1.5B via llama.cpp with a restricted context window
    (2048 n_ctx) and reduced batch size (128 n_batch) for CUDA stability.
  - Key Features: Real-time streaming chat, Agentic Thinking workflows,
    concurrency management for shared GPU access, and human-in-the-loop terminal
    execution.

Class Responsibilities

Frontend (VS Code Extension - TypeScript):

  - chat_view.ts (ChatViewProvider): Manages the Webview UI. Implements
    JSON-stream buffering (protection against fragmented chunks), passes the
    workspace directory path to the backend, and handles the AbortController for
    safe generation interruption.
  - ghost_text.ts (GhostTextProvider): Provides inline code completions.
    Operates via the backend's request queue, functioning alongside the chat.
  - Webview Script: Handles real-time markdown rendering and interactive UI
    elements like Accept/Reject buttons for terminal commands.

Backend (FastAPI - Python):

  - llm.py (LLMEngine): A wrapper for llama_cpp. Configured for thread-safe
    operations (atomic/streaming). Optimized for low-memory environments via
    n_batch and n_gpu_layers parameters.
  - orchestrator.py (Orchestrator): The central logic controller. Manages GPU
    access via asyncio.Lock, preventing CUDA Access Violation errors. Implements
    self-correction cycles and multi-stage planning.
  - planner.py (TaskPlanner): Deconstructs complex coding tasks into logical
    steps, minimizing model "hallucinations" through context segmentation.
  - tools.py (ToolExecutor): Executes terminal commands within the user's
    specific workspace_path context.

Current State

An Agentic Reasoning Pipeline is implemented, allowing the 1.5B model to solve
complex tasks through self-critique:

1.  Concurrency Safety (GPU Locking): Implemented a locking mechanism
    (asyncio.Lock). Requests from Ghost Text and Chat are queued, preventing
    CUDA crashes during simultaneous VRAM access.
2.  Fast Mode: Direct response generation for simple questions and quick
    instructions.
3.  Thinking Mode (Recursive Chain-of-Thought):
      - Goal Analysis: The model first formulates a concise goal to maintain
        "focus" throughout the entire cycle.
      - Iterative Critique: Every response is verified (logically or via
        ast.parse). If the result fails to meet the goal, the error reason is
        fed back into the next iteration (up to 3 attempts).
      - Code Assembly: For coding tasks, the system generates fragments based on
        a plan. A final "Architect" stage assembles these fragments into a
        single file, removing duplicate imports and functions.
      - Attempt Transparency: The UI displays all reasoning cycles and attempts
        separated by headers, allowing the user to compare different versions.
4.  Robust Streaming: The frontend utilizes a text buffer to collect fragmented
    JSON packets, ensuring smooth text rendering even during unstable generation
    speeds.
5.  Safe Terminal: Commands are generated based on environment constraints,
    displayed for user verification, and executed strictly within the project's
    working directory.

Testing System (Benchmarks)

An automated sandbox designed to verify architectural stability and reasoning
quality.

  - Structure: Thematic test suites (Algorithms, CLI, Logic).
  - Runner: Creates isolated temp directories, executes generated code, and
    verifies results via assert statements. Supports real-time CSV logging and
    manual mode selection (fast vs. thinking).
  - Metrics: Focuses on the Self-Correction Success Rate (success after
    retries), processing time.
  - Warning Detection: Automatically detects if the AI output contains a
    "Warning" badge, indicating failed internal verification.
  - Running from Docker

Technical Constraints & Solutions

  - VRAM Limit (2GB): Solved by reducing n_ctx to 2048 and n_batch to 128. This
    prevents "Out of Memory" errors during deep reasoning cycles.
  - Thread Safety: Resolved using asyncio.to_thread for the execution of
    synchronous llama_cpp calls and a global asyncio.Lock within the
    Orchestrator, making the local server stable under parallel requests from VS
    Code.
  - Instruction Adherence: Addressed by strictly limiting tokens in planning
    stages and using specialized system prompts to prevent the 1.5B model from
    looping or repeating instructions.
